# AI项目-Harness-总设计-v1

> 用途：定义 AI 时代从 0 到 1 启动项目时，如何把“提示词、项目模板、测试、部署、文档回写、人工闸门”收敛成一套可持续执行的 Harness。

## 一、先说结论

如果你想让 Agent 真正做到“少问、连续推进、自动测试、自动回写、自动部署”，不要只做一段大提示词，也不要先沉迷多 Agent 编排。

更稳的做法是同时具备三层：

1. 方法论与规范层
2. 仓库内 Harness 层
3. 执行平台层

三层里，前两层必须掌握在自己手里；第三层可以后接单 Agent，也可以后接多 Agent。

## 二、三层架构

### 1. 方法论与规范层
这层回答的是：
- 项目目标怎么定义
- MVP 怎么收敛
- 什么算完成
- 哪些节点必须人工拍板
- Agent 每轮执行后必须回写什么

这层建议沉淀在知识库里，作为长期复用的规范资产。

### 2. 仓库内 Harness 层
这层回答的是：
- 这个项目的真实规则写在哪
- 当前版本目标写在哪
- 测试入口写在哪
- 部署方式写在哪
- 文档回写写在哪

这层必须进入项目仓库本身，而不是只存在聊天上下文里。

推荐最小结构：

```text
AGENTS.md
docs/specs/
docs/runbooks/
docs/iterations/
docs/reviews/
```

### 3. 执行平台层
这层回答的是：
- 用单 Agent 还是多 Agent
- 是否并行拆任务
- 谁写代码，谁验收，谁回归
- 谁在关键闸门停下来问人

这层可以按项目复杂度替换，不应反过来绑死前两层。

## 三、为什么不能只靠提示词

只靠提示词，常见问题有：

- 目标在多轮对话里逐步漂移
- 测试标准没有固化，Agent 以为“能跑”就算完成
- 部署和运维规则只在某次聊天出现过，后面丢失
- 文档回写没有机制，导致每次都重新解释背景
- 多 Agent 并行时没有共同约束，越快越乱

所以真正的 Harness，不是“更强提示词”，而是“把规则落到仓库里”。

## 四、这套 Harness 要解决的核心问题

### 1. 启动问题
新项目开始时，只给一个大方向，Agent 就能先把项目骨架、规则骨架、验证骨架搭起来。

### 2. 连续推进问题
Agent 不要每一步都问，而是在未触发闸门时连续向前推进。

### 3. 验证问题
每轮都必须有可执行验证，而不是只汇报“我做完了”。

### 4. 回写问题
每次迭代后，把目标变化、决策、风险、后续事项沉淀回仓库。

### 5. 交接问题
换一个 Agent、换一个平台、隔一天再继续，都能从仓库里的真实记录接上。

## 五、推荐工作流

### 阶段 A：初始化
输入：
- 项目名称
- 项目目标
- 目标用户
- 交付类型
- 技术栈偏好（可选）

输出：
- 项目 `AGENTS.md`
- 项目启动卡
- MVP 范围与非目标
- 验收标准
- 人工闸门
- 测试与验证策略
- 部署/运维模板
- 首轮迭代卡

### 阶段 B：监督式自动推进
Agent 默认按以下节奏工作：

1. 读取仓库内规范
2. 明确本轮目标
3. 编码
4. 运行测试 / 构建 / 验证
5. 记录结果
6. 判断是否触发人工闸门
7. 未触发则继续下一轮

### 阶段 C：半自主 / 自主推进
当前提成立时，压缩人工介入，仅在以下节点暂停：

- 首次目标定义
- MVP 首次确认
- 第一版可见结果
- 方向变化
- 发布 / 合并 / 对外展示

## 六、单 Agent 与多 Agent 的关系

默认建议：

1. 先让单 Agent 跑通完整闭环
2. 再把实现、测试、审查、验收拆成多 Agent

原因：
- 单 Agent 更容易先稳定规约
- 多 Agent 的价值是并行和交叉审查
- 多 Agent 不是替代规范，而是放大规范执行效率

## 七、推荐产物形态

为了让这套东西能真正复用，建议最终交付不是单一文档，而是三件套：

### 1. 知识库文档
沉淀方法论、决策规则、使用说明。

### 2. 项目模板仓
沉淀仓库内 `AGENTS.md`、spec、runbook、iteration 模板。

### 3. 启动入口
用一个 skill 或一句话母提示词，把模板复制到新项目并填入最小上下文。

## 八、最低可用标准

如果只做最小版，也至少要满足：

1. 新项目能一键初始化出 Harness 骨架
2. 任何 Agent 进入项目先读 `AGENTS.md` 和 `docs/specs/`
3. 每轮迭代都有验证
4. 每轮迭代都回写文档
5. 只在闸门节点才问人

## 九、一句话总结

> AI 时代的项目 Harness，本质上不是“一个万能提示词”，而是“知识库规范 + 仓库内模板 + 可调用启动入口”的组合系统。
