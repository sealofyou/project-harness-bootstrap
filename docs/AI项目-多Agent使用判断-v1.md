# AI项目-多Agent使用判断-v1

> 用途：判断一个项目在什么阶段适合继续用单 Agent，什么阶段值得升级到多 Agent。

## 一、先说结论

不是所有项目都应该一开始就上多 Agent。

推荐顺序：

1. 先用单 Agent 跑通完整闭环
2. 闭环稳定后，再引入多 Agent 放大效率

## 二、什么时候单 Agent 就够了

满足以下多数条件时，优先单 Agent：

- 项目还在 0 到 1 阶段
- 目标和范围还会变
- 技术栈尚未完全确定
- 关键工作还是“搭骨架”和“对齐方向”
- 项目规模较小，任务强依赖上下文连续性

单 Agent 的优势：
- 上下文集中
- 不容易出现并行冲突
- 更适合先把规范和模板做稳

## 三、什么时候值得上多 Agent

满足以下多数条件时，可以考虑多 Agent：

- 前后端可以并行
- 测试、实现、审查可以拆开
- 项目周期长，跨天迭代多
- 你希望把“实现 / 测试 / 验收 / 文档回写”做成流水线
- 已有稳定的仓库内 Harness 作为共同规则

多 Agent 最常见的三个收益：

1. 并行实现
2. 交叉审查
3. 分角色验证

## 四、不适合上多 Agent 的危险信号

如果出现这些情况，先别上多 Agent：

- 项目目标还没写清楚
- 验收标准不明确
- 测试入口没定
- 仓库里没有统一 `AGENTS.md`
- 不同 Agent 进入项目会读到不同版本的规则

此时多 Agent 只会把混乱放大。

## 五、建议的升级路径

### 阶段 1：单 Agent 闭环
完成：
- 启动
- 拆解
- 编码
- 测试
- 验收
- 部署准备
- 文档回写

### 阶段 2：双 Agent 分工
可以先拆成：
- 实现 Agent
- 验证 Agent

### 阶段 3：多 Agent 流水线
进一步拆成：
- 规划 Agent
- 实现 Agent
- 测试 Agent
- 审查 Agent
- 发布 Agent
- 文档回写 Agent

## 六、推荐原则

如果你不确定，就按下面这句执行：

> 先让单 Agent 在 Harness 约束下稳定工作，再让多 Agent 在同一套 Harness 下并行放大。

## 七、一句话总结

> 多 Agent 不是项目 Harness 的起点，而是项目 Harness 稳定后的放大器。
