## 对照结论

PDF 中可复用的方法，和本仓库当前前端提示词体系基本一致。

当前不适合把 PDF 内容整段搬进项目提示词。更合适的做法是继续保留“方法转译”：

- 主 Agent 只读取前端总入口。
- 设计子 Agent 读取设计系统、风格、上下文和验收规则。
- 实现子 Agent 读取页面、组件、状态和浏览器验证规则。
- 测试子 Agent 后续读取交互、视觉和美观度验收规则。

这样能减少主 Agent 上下文污染，也方便后续把不同提示词拆给不同子 Agent 使用。

## PDF 中已经覆盖的点

这些内容已经在 `30-前端-设计与实现.md`、`31-前端-设计系统与风格.md` 和 `32-前端-页面实现与状态.md` 中体现：

- 先收集上下文，再做设计。
- 优先读取现有代码、设计系统、截图、品牌资料和用户提供的参考。
- 缺少关键上下文时先追问，不假装已经完成高保真设计。
- 设计探索应提供多个可区分方向，而不是只换配色。
- 设计交付要能指导实现和验收。
- 实现阶段要读取设计交接，不重新发明设计。
- 完成后要通过浏览器或可执行验证确认结果。
- 不把外部系统提示词原文照搬成本项目规则。

## 本次补充的缺口

这次只补了两个长期有价值的规则：

1. 设计迭代时优先围绕一个主交付物做变体，而不是拆成多个互相独立的散文件。
2. 禁止复刻受版权保护的设计或标志性界面，只提炼结构、节奏、取舍和验收标准。

## 后续处理原则

以后如果继续补 Claude Design / Cloud Design 资料：

- 先判断它是新方法，还是对已有方法的解释。
- 只有能改变工作流、交付格式或验收标准的内容，才进入提示词。
- 解释性内容、来源信息和对照记录放在 `docs/references/`。
- 不为了“看起来完整”扩写提示词。
