# 提示词优化路线图

日期：2026-05-11

## 当前判断

Git 阶段已经足够支撑下一轮迭代：提交标题规范、多 Agent 工作区隔离、主会话集成、跨电脑同步和推送规则都已经进入模板。

任务队列、Linear、多人协作和远程项目管理暂时不需要继续提前展开。它们属于协作层，应该等产品、前端、后端、测试这条本地开发闭环更稳定后再接入。

因此当前应从 Git 转入前端设计，先把设计系统、页面结构、视觉风格和验收标准整理清楚，再进入前端代码实现。

## 优先级顺序

1. 产品 / PRD
2. Git
3. 前端设计
4. 前端实现
5. 后端
6. 测试
7. 协作 / 远程仓库 / Linear / 多 Agent 项目管理

## 为什么这样排

前端和 Git 是最影响日常开发质量的两块。

Git 已经先被整理，是因为之后每一次提示词优化都会产生文档或代码变更。如果 Git 规则不清楚，后面跨电脑同步、回看历史和继续迭代都会变得混乱。

前端现在应优先做设计，而不是直接做代码实现。前端质量不只来自组件代码，也来自产品类型判断、信息密度、页面结构、视觉约束、响应式和验收标准。

## 阶段 1：产品 / PRD 提示词

目标：让项目启动时稳定问出关键问题，并形成可交接的产品事实层。

相关文件：

- `templates/base/docs/prompts/05-启动后交互契约.md`
- `templates/base/docs/prompts/20-产品-需求深访与PRD.md`
- `templates/base/docs/specs/01-项目启动卡.md`
- `templates/base/docs/specs/02-MVP范围与非目标.md`
- `templates/base/docs/specs/03-验收标准.md`

已完成：

- 增加启动后不直接结束、继续提问和交接的规则。
- 增加产品深访、PRD 和 `Product Handoff` 的约束。
- 明确未知信息写入 `待确认`。

后续可继续优化：

- 补充用户旅程、故事卡和版本拆分提示词。
- 接入用户后续提供的产品文章或技能资料。

## 阶段 2：Git 提示词

目标：让每轮变更都易同步、易审查、可回退、可继续。

相关文件：

- `templates/base/docs/prompts/50-Git-仓库维护.md`
- `docs/runbooks/git-sync-and-contribution.md`
- `templates/base/AGENTS.md`

已完成：

- 规定中文 Conventional Commit 标题。
- 补充提交正文中的约束、取舍、验证和未测项。
- 补充跨电脑同步和推送说明。
- 补充多 Agent 并行开发时的 worktree 隔离、主会话集成和子 Agent 禁止自行提交推送。

暂缓：

- Linear / issue / 版本管理暂时不提前复杂化。
- 自动化任务队列暂时不接入 Git 阶段。

## 阶段 3：前端设计提示词

目标：让前端工作从产品意图和设计质量出发，而不是随机生成组件。

相关文件：

- `templates/base/docs/prompts/30-前端-设计与实现.md`
- `templates/base/docs/prompts/31-前端-设计系统与风格.md`
- `templates/base/docs/prompts/32-前端-页面实现与状态.md`
- `templates/base/docs/prompts/60-测试-交互视觉与验收.md`

已完成：

- 将 `30-前端-设计与实现.md` 改成前端总入口。
- 新增 `31-前端-设计系统与风格.md`，专门负责设计系统、视觉风格、页面结构和设计验收。
- 在 `AGENTS.md` 和默认协作流程中挂载设计系统提示词。
- 参考 Claude Design 公开生态资料，强化设计上下文收集、多方向探索、设计推理和实现交接。
- 新增 `32-前端-页面实现与状态.md`，专门负责组件实现、状态覆盖、响应式和浏览器验证。

下一步：

- 结合用户后续提供的 Cloud Design / Claude Design PDF，继续升级设计系统与实现提示词。
- 继续补前端测试验收规则，尤其是截图、移动端、视觉偏差和设计达标检查。

## 阶段 4：后端提示词

目标：让后端工作遵守产品规则和接口契约，不凭空补业务逻辑。

相关文件：

- `templates/base/docs/prompts/40-后端-接口与数据.md`

计划：

- 拆清 API 契约、数据模型、权限、后台任务、外部服务和错误处理。
- 增加 `.env.example` 和运行手册更新要求。
- 增加给前端的接口交接格式。
- 增加最小后端验证清单。

## 阶段 5：测试提示词

目标：让测试真正发现问题，而不是只确认 happy path。

相关文件：

- `templates/base/docs/prompts/60-测试-交互视觉与验收.md`
- `templates/base/docs/specs/05-测试与验证策略.md`

计划：

- 继续拆分工程回归、交互点击、普通用户体验、视觉美观度、设计达标验收。
- 增加前端项目的截图验证要求。
- 增加问题报告到修复任务的交接格式。
- 增加“修复后由测试角色复验”的规则。

## 阶段 6：协作 / Linear / 远程项目管理

目标：在本地开发闭环稳定之后，再接入协作层。

相关文件：

- `templates/base/docs/prompts/70-协作-远程仓库与Linear.md`
- 后续可选：`templates/base/docs/prompts/71-Linear-Issue与版本管理.md`
- 后续可选：`templates/base/docs/prompts/72-多Agent任务拆分.md`

计划：

- 定义什么项目值得接入 Linear。
- 定义 feature、bug、test、docs、research 的 issue 模板。
- 定义 branch / issue / PR 的关联方式。
- 保持 repo docs 是事实来源，Linear 只管理协作状态。
