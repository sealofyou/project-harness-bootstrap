# 模板专用化设计

## 背景

当前仓库只有一个通用 Harness 模板，能够初始化项目规约，但还不能根据用户意图自动选“纯前端”或“前端 + FastAPI”模板。

用户希望：

1. `main` 保持通用主线
2. 模板按目录而不是长期分支维护
3. 后续 AI 通过自然语言自动选择模板
4. `frontend-fastapi` 模板默认具备基础日志与可读注释
5. Python 环境优先 `conda`，没有时回退到 `uv`

## 目标

在不破坏通用 Harness 的前提下，增加两套专用模板：

1. `frontend-only`
2. `frontend-fastapi`

并让初始化脚本可以：

1. 显式指定模板
2. 根据用户自然语言自动推断模板
3. 在未明确技术栈时仅初始化 `base`

## 结构方案

采用目录分层而不是分支分层：

```text
templates/
  base/
  frontend-only/
  frontend-fastapi/
```

初始化逻辑：

1. 先复制 `base`
2. 再按选中的模板叠加覆盖层

## `frontend-only` 设计

只补前端专属差异：

- 前端目录建议
- 前端验证命令占位
- 前端部署说明
- 前端验收与截图要求

不引入后端结构。

## `frontend-fastapi` 设计

在 `base` 之上增加：

- `frontend/`
- `backend/`
- FastAPI 最小目录结构
- 环境初始化脚本
- 日志配置文件
- 请求级与关键流程日志
- 最小 API 示例
- 测试目录与 runbook

## Python 环境策略

优先级：

1. `conda`
2. `uv`
3. 二者都没有时保留占位并在 runbook 中说明

## 日志策略

日志必须覆盖：

1. 应用启动
2. 请求进入与完成
3. 关键业务流程
4. 异常处理

日志不追求铺满全项目，但要保证排查入口清晰。

## 注释策略

使用简短、解释性的注释：

- 配置与初始化处
- 关键边界判断处
- 复杂流程入口处

避免无意义注释。

## 自动模板选择

规则：

1. 明确提到纯前端、官网、前端页面时，默认 `frontend-only`
2. 明确提到 FastAPI、Python 后端、前后端分离 API 时，默认 `frontend-fastapi`
3. 未明确时仅使用 `base`

## 验证方式

通过脚本测试验证：

1. `base` 初始化
2. `frontend-only` 初始化
3. `frontend-fastapi` 初始化
4. 自动意图识别
5. 关键文件存在
6. 关键文案与模板内容落位
