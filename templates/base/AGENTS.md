# AGENTS.md - Project Harness

本文件是当前项目内所有 Agent 的总规约。

## 一、目标摘要

- 项目名称：__PROJECT_NAME__
- 项目目标：__PROJECT_GOAL__
- 目标用户：__TARGET_USER__
- 技术栈偏好：__TECH_STACK__

1. `AGENTS.md`
2. `docs/specs/01-项目启动卡.md`
3. `docs/specs/02-MVP范围与非目标.md`
4. `docs/specs/03-验收标准.md`

9. `docs/prompts/05-启动后交互契约.md`
10. 当前正在执行的 `docs/iterations/*.md`

## 三、默认执行流程

1. 先复述当前轮目标
2. 先遵守 `docs/prompts/01-系统级规则.md`
3. 如果项目刚初始化完成，先执行 `docs/prompts/05-启动后交互契约.md`
4. 判断本轮范围是否明确
5. 若不明确，先使用 `docs/prompts/20-产品-需求深访与PRD.md` 澄清，不直接实现
6. 若明确，按任务类型选择 `docs/prompts/` 下的专门提示词
7. 实现后必须执行约定验证
8. 将结果回写到迭代文档
9. 判断是否触发人工闸门
10. 未触发则继续下一轮

## 四、提示词调用边界

`docs/prompts/` 是本项目的默认提示词资产层。

- 系统级规则：`docs/prompts/01-系统级规则.md`
- 系统流程：`docs/prompts/10-系统-默认协作流程.md`
- 启动后交互：`docs/prompts/05-启动后交互契约.md`
- 产品 / PRD：`docs/prompts/20-产品-需求深访与PRD.md`
- 前端 / UI / 交互实现：`docs/prompts/30-前端-设计与实现.md`
- 后端 / API / 数据：`docs/prompts/40-后端-接口与数据.md`
- Git / 分支 / 提交 / PR：`docs/prompts/50-Git-仓库维护.md`
- 测试 / 交互 / 视觉 / 美观度验收：`docs/prompts/60-测试-交互视觉与验收.md`
- GitHub remote / Linear / 协作：`docs/prompts/70-协作-远程仓库与Linear.md`

原则：一个子 Agent 只承担一个专门角色。不要让同一个子 Agent 同时负责需求、实现、测试和 Git 收尾。

项目刚初始化完成时，不要只报告“文件已创建”。必须列出已确认 / 待确认信息，并向用户提出一个最关键问题。

所有 Agent 默认使用中文。除代码、命令、API、包名、配置键、环境变量、框架名、专有名词等必要情况外，文档名称、文档正文、提交标题说明、PR 说明、测试报告都必须使用中文。

## 五、人工闸门

遇到以下情况必须暂停：

- 首次目标定义或方向变更
- MVP 范围扩大
- 第一版可见结果已出现
- 发布、合并、对外展示前
- 存在两个以上都合理但偏好差异明显的方向

## 六、验证要求

在声称“完成、通过、可发布”之前，必须：

1. 运行本项目定义的测试或验证命令
2. 阅读结果
3. 在迭代文档中记录验证证据

不能只说“应该可以”。

## 七、文档回写要求

每轮至少回写：

- 本轮目标
- 本轮改动
- 验证结果
- 风险 / 未决项
- 下一轮建议

如果发生范围变化，还要同步更新 `docs/specs/`。

## 八、默认原则

- 默认连续推进，不要做一步问一步
- 未触发闸门时，允许小步快跑
- 优先删除而不是新增复杂层
- 优先复用现有模式
- 未知信息写成待确认，不要直接丢失
