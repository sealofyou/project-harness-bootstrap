# 31-前端-设计系统与风格

## 定位

本文件专门负责前端设计系统、视觉风格、页面信息结构和设计验收标准。

它应在前端代码实现之前使用。目标不是写漂亮形容词，而是把“这个产品应该长什么样、为什么这样设计、后面怎么验收”说清楚。

本文件参考 Claude Design 公开生态中可复用的方法：先收集上下文和设计资源，再做设计探索；优先用设计系统、UI kit、截图、品牌资料和现有代码约束输出；缺少上下文时先追问；交付时形成可被实现 Agent 和测试 Agent 继续使用的设计交接，而不是只给一张“看起来不错”的图。

## 触发时机

遇到以下情况，先使用本提示词：

- 用户要求做页面、应用、网站、后台、工具、组件或交互。
- 项目缺少设计系统、视觉方向、页面结构或风格约束。
- 用户对“好看”“高级”“简洁”“专业”“可爱”“有设计感”等表达不够具体。
- 前端实现前，需要明确组件样式、布局、响应式和视觉验收标准。
- 后续要让子 Agent 分别实现页面，需要先给它们统一设计约束。

## 必须确认的信息

设计前至少确认这些信息；缺失时只问一个最关键问题：

- 目标用户：谁用，使用场景是什么。
- 产品类型：工具、SaaS、后台、内容站、作品集、电商、游戏或其他。
- 核心任务：用户最重要的一条完成路径。
- 情绪和气质：冷静、专业、轻盈、可信、活泼、实验性等。
- 信息密度：工作台式高密度，还是展示型低密度。
- 参考材料：截图、Figma、品牌规范、竞品、文章、PDF、Cloud Design 等。
- 设备范围：移动端、桌面端、平板、宽屏是否都要覆盖。
- 可访问性要求：对比度、键盘操作、字号、动效敏感等。

如果用户还没有参考材料，不要编造来源。可以先产出临时设计方向，并标注 `待确认`。

## 设计流程

### 0. 设计上下文收集

开始设计前，先寻找已有设计上下文。

优先级如下：

1. 目标项目已有代码、组件、样式、设计 token。
2. `DESIGN.md`、品牌规范、截图、Figma、PDF、竞品页面。
3. 用户提供的颜色、字体、文案、行业参考。
4. 公开资料或相近产品模式。
5. 临时设计假设。

如果前 3 类都没有，不要直接进入高保真设计。先问用户一个关键问题，例如：

- 是否有参考产品或截图？
- 希望更接近工具型、内容型还是品牌型？
- 需要几个方向的视觉探索？

如果用户暂时没有材料，可以继续，但必须在设计交接中标注：`设计上下文不足，当前为临时方向`。

### 1. 判断产品类型

先判断产品更接近哪一类：

- 工作型工具：优先效率、密度、扫描、重复操作。
- SaaS / CRM / 管理后台：优先清晰层级、表格、筛选、状态反馈。
- 内容或品牌页面：优先叙事、第一屏信号、可读性和真实视觉素材。
- 创作工具：优先画布、工具栏、快捷操作和状态可见性。
- 游戏或互动体验：优先沉浸感、反馈、节奏和可玩性。

不同类型不能共用一套“营销式大卡片”方案。

### 2. 选择探索策略

设计不应只给一个方向。除非用户明确要求只要一版，否则至少提供 2 到 3 个可区分方向：

- 保守方向：严格贴近现有组件、行业惯例和可实现性。
- 推荐方向：在一致性和视觉辨识度之间取得平衡。
- 探索方向：尝试更有记忆点的布局、节奏、动效或视觉隐喻。

每个方向都要说明：

- 适合什么用户和场景。
- 牺牲了什么。
- 对实现成本有什么影响。
- 是否适合当前 MVP。

不要把三个方向做成同一布局换三套颜色。

如果需要持续迭代，优先围绕一个主交付物做变体，而不是把多个版本拆成互相独立的散文件。主版本负责承载设计决策，变体负责承载局部调整，这样更容易让主 Agent、前端实现 Agent 和测试 Agent 接力。

### 3. 建立设计方向

输出设计方向时，必须具体到可执行约束：

- 布局：页面分区、导航方式、主要内容区域、次要信息区域。
- 色彩：主色、辅助色、背景、边框、状态色和禁用色。
- 字体：标题、正文、数字、标签、按钮的层级。
- 间距：页面边距、区块间距、控件间距、列表密度。
- 控件：按钮、输入框、选择器、菜单、标签、表格、弹窗。
- 状态：hover、focus、active、disabled、loading、empty、error、success。
- 动效：只使用服务理解和反馈的微动效，避免干扰。
- 图标：优先使用一致的图标库；不要用 emoji 充当 UI 图标。
- 资产：真实图片、品牌资源、图标、插画、占位资源的来源和替代策略。

### 4. 写出设计推理

设计交付物必须包含简短设计推理。

推理只说明必要取舍：

- 为什么这种信息结构适合当前用户路径。
- 为什么这种密度适合当前产品类型。
- 为什么选择这些颜色、字体、空间和控件。
- 哪些地方是临时假设，后续需要用户或真实数据确认。

不要写大段抽象审美描述。每条推理都要能影响实现或验收。

### 5. 形成设计交付物

如果本次工作会进入实现阶段，应创建或更新：

- `docs/design/设计系统.md`
- `docs/design/页面结构.md`
- `docs/design/视觉验收清单.md`
- `docs/design/设计探索记录.md`

内容可以先轻量，但必须能指导后续子 Agent 实现和测试。

### 6. 给实现 Agent 的交接

交接给实现阶段时，输出 `前端设计交接卡`：

```markdown
# 前端设计交接卡

## 产品类型

## 目标用户

## 核心路径

## 设计方向

## 页面结构

## 设计系统约束

## 响应式要求

## 可访问性底线

## 视觉验收重点

## 选中方向与未选方向

## 关键组件清单

## 资产与占位策略

## 待确认
```

只要 `待确认` 中存在会影响布局、风格或核心交互的问题，就不要声称设计已经完成。

## 视觉质量底线

设计方案必须满足：

- 信息层级一眼可扫。
- 主要操作始终明显。
- 文字不依赖超小字号维持布局。
- 普通文本对比度足够。
- 移动端不横向滚动。
- 点击目标不小于常见移动端可点击尺寸。
- 焦点态可见，键盘可以走完关键路径。
- 加载、空状态、错误状态不破坏布局。
- 色彩不是唯一状态提示。

## 参考资料处理

如果用户提供文章、PDF、截图、Figma 或竞品链接：

- 先读资料，再提炼可执行规则。
- 不要照抄视觉外壳，要提炼结构、节奏、取舍和验收标准。
- 对不确定的资料结论标注 `待确认`。
- 重要结论回写到 `docs/design/`。

如果用户提供 Claude Design / Cloud Design 相关 PDF 或导出的 `DESIGN.md`：

- 先完整阅读资料。
- 提炼其中的设计流程、上下文收集方式、输出格式和验收规则。
- 把适合长期复用的内容沉淀到 `docs/design/` 或本提示词。
- 不照搬系统提示词原文，不把外部工具的内部约束当成本项目必须遵守的规则。

## Claude Design 方法转译

使用 Claude Design 类资料时，优先吸收这些方法：

- 设计前先找上下文，不从空白画布开始。
- 好设计来自现有设计系统、组件、品牌和真实使用场景。
- 设计探索要给多个方向，方便用户混合选择。
- 高保真原型可以先用 HTML / CSS / JS 表达，但最终必须能交给工程项目继续实现。
- 设计交付要带可验证规则，而不是只靠主观“好看”。
- 早展示、早反馈、早迭代，比一次性憋完整方案更稳。
- 如果无法定位用户评论对应的代码或组件，先探查再修改，不猜。

这些方法是流程约束，不是要求所有项目都生成单个 HTML 设计稿。

## 禁止事项

- 不要用“高级感”“科技感”“干净”这类词替代具体设计约束。
- 不要在没有目标用户和核心路径时直接选风格。
- 不要把后台、工具类产品做成营销落地页。
- 不要只给单一色相的配色方案。
- 不要让装饰元素压过内容。
- 不要在没有响应式和可访问性约束时交付设计。
- 不要缺少设计上下文时假装已经完成高保真设计。
- 不要复刻受版权保护的设计或标志性界面。
