# 31-前端-设计系统与风格

## 定位

本文件专门负责前端设计系统、视觉风格、页面信息结构和设计验收标准。

它应在前端代码实现之前使用。目标不是写漂亮形容词，而是把“这个产品应该长什么样、为什么这样设计、后面怎么验收”说清楚。

## 触发时机

遇到以下情况，先使用本提示词：

- 用户要求做页面、应用、网站、后台、工具、组件或交互。
- 项目缺少设计系统、视觉方向、页面结构或风格约束。
- 用户对“好看”“高级”“简洁”“专业”“可爱”“有设计感”等表达不够具体。
- 前端实现前，需要明确组件样式、布局、响应式和视觉验收标准。
- 后续要让子 Agent 分别实现页面，需要先给它们统一设计约束。

## 必须确认的信息

设计前至少确认这些信息；缺失时只问一个最关键问题：

- 目标用户：谁用，使用场景是什么。
- 产品类型：工具、SaaS、后台、内容站、作品集、电商、游戏或其他。
- 核心任务：用户最重要的一条完成路径。
- 情绪和气质：冷静、专业、轻盈、可信、活泼、实验性等。
- 信息密度：工作台式高密度，还是展示型低密度。
- 参考材料：截图、Figma、品牌规范、竞品、文章、PDF、Cloud Design 等。
- 设备范围：移动端、桌面端、平板、宽屏是否都要覆盖。
- 可访问性要求：对比度、键盘操作、字号、动效敏感等。

如果用户还没有参考材料，不要编造来源。可以先产出临时设计方向，并标注 `待确认`。

## 设计流程

### 1. 判断产品类型

先判断产品更接近哪一类：

- 工作型工具：优先效率、密度、扫描、重复操作。
- SaaS / CRM / 管理后台：优先清晰层级、表格、筛选、状态反馈。
- 内容或品牌页面：优先叙事、第一屏信号、可读性和真实视觉素材。
- 创作工具：优先画布、工具栏、快捷操作和状态可见性。
- 游戏或互动体验：优先沉浸感、反馈、节奏和可玩性。

不同类型不能共用一套“营销式大卡片”方案。

### 2. 建立设计方向

输出设计方向时，必须具体到可执行约束：

- 布局：页面分区、导航方式、主要内容区域、次要信息区域。
- 色彩：主色、辅助色、背景、边框、状态色和禁用色。
- 字体：标题、正文、数字、标签、按钮的层级。
- 间距：页面边距、区块间距、控件间距、列表密度。
- 控件：按钮、输入框、选择器、菜单、标签、表格、弹窗。
- 状态：hover、focus、active、disabled、loading、empty、error、success。
- 动效：只使用服务理解和反馈的微动效，避免干扰。
- 图标：优先使用一致的图标库；不要用 emoji 充当 UI 图标。

### 3. 形成设计交付物

如果本次工作会进入实现阶段，应创建或更新：

- `docs/design/设计系统.md`
- `docs/design/页面结构.md`
- `docs/design/视觉验收清单.md`

内容可以先轻量，但必须能指导后续子 Agent 实现和测试。

### 4. 给实现 Agent 的交接

交接给实现阶段时，输出 `前端设计交接卡`：

```markdown
# 前端设计交接卡

## 产品类型

## 目标用户

## 核心路径

## 设计方向

## 页面结构

## 设计系统约束

## 响应式要求

## 可访问性底线

## 视觉验收重点

## 待确认
```

只要 `待确认` 中存在会影响布局、风格或核心交互的问题，就不要声称设计已经完成。

## 视觉质量底线

设计方案必须满足：

- 信息层级一眼可扫。
- 主要操作始终明显。
- 文字不依赖超小字号维持布局。
- 普通文本对比度足够。
- 移动端不横向滚动。
- 点击目标不小于常见移动端可点击尺寸。
- 焦点态可见，键盘可以走完关键路径。
- 加载、空状态、错误状态不破坏布局。
- 色彩不是唯一状态提示。

## 参考资料处理

如果用户提供文章、PDF、截图、Figma 或竞品链接：

- 先读资料，再提炼可执行规则。
- 不要照抄视觉外壳，要提炼结构、节奏、取舍和验收标准。
- 对不确定的资料结论标注 `待确认`。
- 重要结论回写到 `docs/design/`。

如果用户后续提供 Cloud Design 相关 PDF，应先完整阅读，再把可复用原则沉淀进本文件或项目设计文档。

## 禁止事项

- 不要用“高级感”“科技感”“干净”这类词替代具体设计约束。
- 不要在没有目标用户和核心路径时直接选风格。
- 不要把后台、工具类产品做成营销落地页。
- 不要只给单一色相的配色方案。
- 不要让装饰元素压过内容。
- 不要在没有响应式和可访问性约束时交付设计。
