# 01-系统级规则

## 定位

本文件是项目内所有 Agent 必须遵守的系统级提示词。

它不是某个阶段的专门提示词，而是所有产品、前端、后端、Git、测试、协作任务的共同上层规则。任何子 Agent 开始工作前，都必须继承这些规则。

## 中文优先规则

默认全部使用中文。

必须使用中文的内容：

- 文档名称
- 文档标题
- 文档正文
- 提交标题中的说明部分
- PR / 变更说明
- 迭代记录
- 复盘记录
- 测试报告
- 用户可见说明

允许保留英文的情况：

- 代码
- 命令
- 文件扩展名
- API 名称
- 包名
- 框架名
- 配置键名
- 环境变量
- 固定协议名称，例如 HTTP、JSON、PRD、MVP、Git、GitHub、Linear
- 已存在的专有名词或品牌名

除上述必要情况外，不要使用英文句子替代中文说明。

## 文档命名规则

新增文档时，文件名应使用中文，必要时保留少量英文缩写或专有名词。

推荐：

```text
docs/specs/01-项目启动卡.md
docs/prompts/20-产品-需求深访与PRD.md
docs/runbooks/本地开发说明.md
docs/reviews/2026-05-11-视觉验收复盘.md
```

不推荐：

```text
docs/specs/project-start.md
docs/prompts/frontend-implementation.md
docs/runbooks/local-dev.md
docs/reviews/review-notes.md
```

如果第三方工具强制要求英文文件名，可以使用英文，但要在文档正文中说明原因。

## 有问题必须追问

当出现以下情况时，必须追问或写入 `待确认`：

- 用户目标不清楚
- 目标用户不清楚
- MVP 范围不清楚
- 验收标准不清楚
- 技术栈不清楚
- 数据来源不清楚
- 权限、部署、账号、成本、合规等存在风险
- 有两个以上合理方向，但偏好差异明显

不要把未确认信息当成默认选择。

## 追问方式

- 一次只问一个最关键问题。
- 先说明为什么这个问题重要。
- 用户回答后，先复述理解并确认。
- 确认后再进入下一步。

## 回写规则

重要事实不能只留在聊天里。

需要回写：

- 用户确认的目标
- MVP 范围
- Out-of-Scope
- 技术约束
- 验收标准
- 重要决策
- 风险和未决项

优先回写位置：

- `docs/specs/`
- `docs/iterations/`
- `docs/reviews/`
- 与任务相关的 `docs/prompts/`

## 禁止事项

- 不要用英文长段落写项目文档。
- 不要新增英文文档名，除非工具或生态要求。
- 不要为了显得完整而补用户没确认的信息。
- 不要在需求不清楚时直接实现。
- 不要把聊天记录当成长期事实来源。
