# 05-启动后交互契约

## 用途

本提示词用于项目刚初始化完成后的第一轮交互。

它的目标是防止 Agent 只生成文件就停下，或者直接跳到实现。初始化完成后，Agent 必须主动进入项目理解和需求深访，把“接下来怎么做”变成可执行事实。

## 启动后必须做

当项目刚通过 Harness / Hermes / 初始化脚本创建完成后，主 Agent 必须：

1. 读取 `AGENTS.md`。
2. 读取 `docs/specs/01-项目启动卡.md`。
3. 读取 `docs/specs/02-MVP范围与非目标.md`。
4. 读取 `docs/specs/03-验收标准.md`。
5. 读取 `docs/prompts/00-提示词索引.md`。
6. 读取 `docs/prompts/20-产品-需求深访与PRD.md`。
7. 列出当前已经确定的信息和 `待确认` 信息。
8. 对照产品提示词中的“讨论结束检查清单”，判断哪些信息缺失。
9. 向用户提出第一个最关键的问题。

## 第一个问题怎么选

优先问会影响后续所有实现的问题：

- 如果目标用户不明确，先问目标用户。
- 如果痛点或为什么要做不明确，先问核心痛点。
- 如果第一版范围不明确，先问第一版必须验证什么。
- 如果产品形态不明确，先问交付物是什么。
- 如果前端风格会影响实现，先问目标气质和参考风格。
- 如果技术栈会影响目录和验证，先问是否已有技术约束。

一次只问一个问题。

## 与用户交互方式

用户可以直接用自然语言补充想法。Agent 要负责整理，不要求用户按模板填写。

推荐用户这样说：

```text
我想做一个项目，大概目标是……
第一版我希望先做到……
目标用户是……
我暂时不确定的是……
```

Agent 每轮应该输出：

- 当前理解
- 已确认事项
- 待确认事项
- 一个最关键的问题

## 什么时候可以开始实现

只有满足以下条件后，才能进入实现：

- MVP 范围已经明确。
- Out-of-Scope 已经明确。
- 至少有一条核心用户路径。
- 关键边缘情况已经讨论。
- 验收标准可以被测试。
- 技术栈或项目形态足够明确。
- 当前迭代文档已经记录本轮目标。

## 禁止事项

- 不要初始化完成后直接结束，不做下一步交互。
- 不要需求不清时直接写代码。
- 不要一次问一串问题。
- 不要把 `待确认` 当成默认选择。
- 不要把用户口头想法留在聊天里不回写文档。
