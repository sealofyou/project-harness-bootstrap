# 32-前端-页面实现与状态

## 定位

本文件专门负责把前端设计交接落成页面、组件、状态和真实浏览器体验。

它不负责重新定义产品需求，也不负责重选设计方向。实现前必须先读取：

1. `docs/prompts/01-系统级规则.md`
2. `docs/prompts/30-前端-设计与实现.md`
3. `docs/prompts/31-前端-设计系统与风格.md`
4. `docs/specs/`
5. `docs/design/` 中已有设计交接

如果没有前端设计交接，先回到 `31-前端-设计系统与风格.md`。

## 输入要求

开始实现前必须确认：

- 产品目标和 MVP 范围已经明确。
- 核心用户路径已经明确。
- 页面结构已经明确。
- 设计系统或临时设计约束已经明确。
- 技术栈和现有组件库已经明确；未知则写为 `待确认`。
- 需要覆盖的视口已经明确。
- 本轮实现范围已经明确。

如果缺失项会影响实现，不要自行脑补，先问一个关键问题或交回主 Agent。

## 实现顺序

### 1. 读现有项目

先检查：

- 路由结构。
- 页面目录。
- 组件目录。
- 样式系统。
- 图标库。
- 状态管理。
- 数据请求方式。
- 测试或浏览器验证方式。

优先复用现有模式，不为了视觉效果引入新技术栈。

### 2. 拆页面与组件

每个页面先拆成：

- 页面骨架：导航、主内容、辅助区、操作区。
- 数据区域：列表、表格、详情、图表、卡片。
- 输入区域：表单、筛选、搜索、上传、编辑器。
- 反馈区域：提示、错误、成功、进度、确认。
- 空间结构：页面边距、区块间距、滚动区域、固定区域。

组件边界要服务复用和可读性，不要为了抽象而抽象。

### 3. 覆盖状态

每个可交互页面至少考虑：

- 初始状态。
- 加载状态。
- 空状态。
- 错误状态。
- 禁用状态。
- 成功状态。
- 数据较多时的状态。
- 数据较少时的状态。
- 网络较慢或操作重复点击时的状态。

状态要稳定，不要让页面因为提示文字、加载动画或错误信息发生明显跳动。

### 4. 落实视觉约束

实现时必须对照设计交接检查：

- 色彩是否使用设计系统中的角色，而不是随手取色。
- 字体层级是否与内容层级一致。
- 间距是否稳定。
- 控件尺寸是否适合点击。
- 图标是否来自同一套图标体系。
- 空状态、错误状态、加载状态是否有一致视觉语言。
- 移动端是否不溢出、不重叠、不横向滚动。

如果设计交接缺少某项实现细节，可以做低风险补全，但必须回写为实现决策。

### 5. 浏览器验证

完成实现后必须验证真实页面。

优先检查：

- 主要路径能否完成。
- 按钮、输入、菜单、弹窗、筛选、提交是否可用。
- 加载、空、错、成功状态是否出现。
- 桌面端布局是否稳定。
- 移动端布局是否稳定。
- 控件 hover / focus / disabled 是否清楚。
- 文字是否溢出。
- 页面是否出现横向滚动。

如果项目支持浏览器截图，保留关键截图或截图路径。

## 实现交接格式

实现完成后输出 `前端实现交接卡`：

```markdown
# 前端实现交接卡

## 本轮实现范围

## 修改文件

## 页面与组件

## 已覆盖状态

## 响应式检查

## 浏览器验证

## 与设计交接的偏差

## 待测试重点

## 未测项
```

## 与测试 Agent 的交接

交给测试 Agent 时必须说明：

- 哪些路径必须点击。
- 哪些状态必须复现。
- 哪些视口必须检查。
- 哪些地方最可能有视觉偏差。
- 哪些地方只是临时实现。

测试 Agent 使用 `60-测试-交互视觉与验收.md` 独立验收。

## 禁止事项

- 不要没有设计交接就直接实现高保真页面。
- 不要把所有信息都塞进卡片。
- 不要用不可维护的局部样式堆出视觉效果。
- 不要为了页面好看破坏业务边界。
- 不要只实现 happy path。
- 不要只在桌面端验证。
- 不要在没有浏览器证据时声称页面可用。
