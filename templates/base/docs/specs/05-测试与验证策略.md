# 05-测试与验证策略

## 一、验证目标

每轮至少证明三件事：

1. 核心功能逻辑正确
2. 工程状态可运行 / 可构建
3. 本轮没有明显回归

## 二、推荐验证层级

### 1. 快速验证

- lint
- typecheck
- build

### 2. 功能验证

- 单元测试
- 集成测试
- 接口测试

### 3. 交互点击验证

- 打开关键页面
- 点击关键按钮
- 输入和提交表单
- 切换状态或筛选条件
- 刷新后检查状态是否保留或正确恢复

### 4. 普通用户体验验证

- 只用页面和正常点击判断，不读代码、不调接口、不操作数据库
- 检查页面标题、按钮、表单、空状态、错误状态是否让普通用户看得懂
- 记录疑惑点、卡住点和流程中断点

### 5. 视觉美观度验证

- 检查信息层级、间距、字体、颜色、按钮状态是否专业一致
- 检查是否有重叠、溢出、错位、文本被遮挡
- 检查移动端 `375x812` 视口

### 6. 设计达标验收

- 如果项目有 `DESIGN.md`、截图、Figma、参考站点或品牌规范，必须逐条对照
- 如果没有设计资产，至少对照目标用户、目标气质和核心工作流

### 7. 页面 / 体验留证

- 浏览器手动检查
- Playwright 或同类自动化验证
- 关键页面截图留证

## 四、执行规则

1. 修改前先判断本轮需要覆盖的验证范围
2. 修改后必须至少运行与改动相关的验证
3. 声称“完成”前必须读取实际输出
4. 验证结果要回写到迭代文档
5. 涉及页面或体验时，优先参考 `docs/prompts/60-测试-交互视觉与验收.md`

## 五、发布前最小验证集合

- 构建通过
- 关键路径验证通过
- 无阻塞性错误
- 部署步骤至少走通一次
